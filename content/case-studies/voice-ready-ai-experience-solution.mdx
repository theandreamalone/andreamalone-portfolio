## The voice interaction system

**I proposed splitting the experience into two first-class modes** — Immersive
(voice-led) and Traditional (text-led) — rather than shipping voice as a
secondary feature bolted onto a chat window. Users bring different comfort
levels to AI, and defaulting everyone into one mode punishes whichever group
didn't get it. Traditional mode keeps a persistent thread history in the
sidebar, so people can return to prior answers — important in a telecom
support context where users frequently revisit account or troubleshooting
detail. Immersive mode leads with the voice interface itself: a full
conversational state cycle, not a single "tap to talk" gesture, spanning
listening, stopped listening, a transition state, AI working, AI speaking,
stopped speaking, and back to listening. Each state is a distinct, designed
screen — not a generic spinner standing in for "processing." That distinction
matters in an incident-response context, where a user needs to know the
system heard them correctly before it commits to anything. An audio-reactive
glow reinforces that feedback in real time, removing the "did it hear me?"
uncertainty that undermines trust in voice UX generally.

Input mode stays a deliberate, reversible choice within either mode, not a
fork a user gets locked into. The mic and keyboard exist as toggled states
within the same input bar — someone can start speaking, see a live "Start
speaking" prompt, and drop back to typing mid-conversation without losing
context.

## Holding both modes to parity on mobile

**I proposed carrying both modes through to mobile at full parity**, rather
than letting mobile default to whichever mode compressed more easily onto a
smaller screen. Traditional mode keeps a single-column, thumb-reachable
layout on mobile, prioritizing message legibility. Immersive becomes the
primary mobile interaction — a deliberate choice, since voice suits
on-the-go, hands-busy use better than typing does. The listening state reuses
the same amplitude-reactive glow language as desktop, so the system-status
metaphor doesn't change from one breakpoint to the next. This moved the
mobile build from "a desktop design squeezed onto a phone" to a systematic
responsive approach applied to both modes.

## The response architecture behind it

The assistant's responses aren't generated freeform — they're assembled
through a defined pipeline: a question is classified by intent, mapped to a
required response schema, and built from required "core" content plus
additional content that only appears when specific conditions are met
(uncertainty, conflicting signals, regulatory exposure, and similar named
triggers). A human review gate sits in that pipeline, keyed to a specific,
named list of conditions — high impact, supporting signals below a set
threshold, conflicting signals present, regulatory or safety exposure,
budget or personnel impact — not a general "escalate if unsure" rule.

When that gate flags a response, the signal currently reaches the user through
an expansion component — an inline, expandable explanation attached to the
response, so a user can pull up the reasoning behind a review flag without
leaving the thread. I scoped several other places that same signal could live,
since not every review case belongs in front of the end user: a visual badge
directly on the response ("[X] review recommended"), an internal log entry
visible to supervisors but hidden from the end user, a flag in a review queue
checked after the fact, a metadata tag for audit and compliance tracking, or a
note in the system's own trace. Only the expansion component has shipped; the
rest are scoped, not built.

## Sequenced from explanation toward autonomy

**I sequenced the AI experience from explanation to recommendation to
action.** The first priority was a consistent response structure, exposed
evidence, and defined review requirements — before any higher-autonomy
behavior. Automated actions, reduced human review, and cross-workflow
orchestration belong later in the sequence, only after the system
demonstrates that its reasoning is understandable, reviewable, and
reliable. This was the sequencing logic I established for the response
system; it treats autonomy as something the product earns, not a starting
feature.

## Catching a classification error before it shipped

Early intent classification worked off question *form* — matching who/what/when
phrasing directly to a response type. Review surfaced that this was the wrong
unit: the same phrasing (a "what" question, for example) could legitimately need
different response types depending on what was actually being asked. The
classification approach was corrected to route on intent rather than surface
wording — a change that avoided misrouting responses in production.

## Two decisions that changed the core chat experience

Designing the voice-first experience surfaced two structural changes that were
carried back into the standard chat experience, not just the voice surface:

- Chat was separated from the dashboard entirely, so it now behaves as its own
  standalone destination — the same treatment given to notifications.
- The history panel was redesigned to span the full width of the screen with a
  text link instead of an icon, making past sessions easier to scan and re-enter.

## Accessibility

An accessibility review is in progress, covering landmark structure, heading
hierarchy, reading and focus order, touch target sizing, and color-blindness
simulation across the core voice states. It has already caught one real
issue — a touch target sized at 32×32px, below the 44×44px recommendation —
now queued for a fix.
